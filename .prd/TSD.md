# Spec Design MCP TSD Overview

## 1. 文档目的

本文件用于定义 Spec Design MCP 的技术总览、版本分层和实施原则，不再直接承担 `v0` 的逐项落地规范。

从本次调整开始，技术文档拆分为两层：

- 总览层：说明架构原则、系统边界、版本演进
- 执行层：定义当前版本可直接开工的实现方案

当前可开发基线以 `TSD_V0.md` 为准。

## 2. 技术定位

Spec Design MCP 的首要形态仍然是一个 MCP Server，而不是人类优先的可视化设计器。

系统目标是把上游非结构化设计需求转成结构化页面设计产物，并对下游编码 Agent 输出可消费的交付包。

- `TSD.md`

## 4. 核心技术原则

### 4.1 必须保留的原则

以下原则在所有版本中保持不变：

1. `DesignDOMAST` 是设计真源。
2. 预览与导出都必须由 AST 派生。
3. 所有关键中间产物必须可校验。
4. 确认前看预览，确认后看交付包。
5. 工具协议必须结构化、可追踪、可恢复。

### 4.2 本轮新增原则

为适应 vibe coding 落地，新增以下约束：

1. 先减少可变性，再扩展能力。
2. 首版优先本地可跑，而不是平台级完备。
3. 能用同步流程解决的问题，不在首版引入异步子系统。
4. 能用严格 schema 解决的问题，不依赖提示词隐式约定。

## 5. 版本分层

### 5.1 V0 技术目标

`v0` 需要具备以下能力：

- MCP 工具可调用
- 单页设计会话闭环
- 严格受限的 AST 生成与修订
- 本地预览产物生成
- 本地导出包生成
- 基础版本控制与确认机制

### 5.2 V0 明确不做

以下能力明确后置：

- 图片输入主链路
- 复杂 URL 抓取治理
- 临时资源签名 URL
- Cleanup Worker
- 对象存储
- 大规模多 Agent 并发策略

### 5.3 V1+ 再引入的系统复杂度

待 `v0` 验证后，再考虑接入：

- PostgreSQL
- S3 兼容对象存储
- Preview TTL / signed URL
- 资源回收任务
- 更复杂的节点类型和页面类型

## 6. 当前推荐技术路径

对于 `v0`，推荐采用低门槛实现路线：

- 运行时：Node.js + TypeScript
- MCP 层：官方 MCP SDK
- 校验层：Zod 作为单一契约源，必要时生成 JSON Schema
- 持久层：SQLite 或文件型存储
- 产物存储：本地文件系统
- 预览截图：Playwright

说明：

- PostgreSQL 和对象存储不再作为 `v0` 强依赖
- 若预览截图实现成本过高，可先保留 `preview.html`，截图作为增强项

## 7. 模块收敛策略

`v0` 不再强制拆成过多独立模块，建议收敛为以下几个实现单元：

- `mcp-server`
- `conversation-service`
- `ast-service`
- `preview-service`
- `export-service`
- `storage-service`

只有在代码量和职责已经明显膨胀时，才继续拆包。

## 8. 关键技术调整

### 8.1 AST 收敛

`v0` 的 AST 必须收敛为有限节点集合、有限布局模型、有限样式字段，避免“生成端随意、编译端兜底”。

### 8.2 写操作显式版本化

以下写操作都必须引入版本前置条件：

- 修订
- 确认
- 导出

避免多 Agent 或重复调用导致版本漂移。

### 8.3 预览先本地化

`v0` 的预览能力以本地文件和可选截图为主，不把签名 URL、TTL、回收机制作为上线前提。

### 8.4 导出契约先做稳定

`artifact-manifest.json`、`compiled.html`、`compiled.css`、`annotation-manifest.json`、`binding.schema.json` 需要先保证字段稳定、引用关系明确，再讨论更复杂的语义扩展。

## 9. 实施顺序

推荐按照以下顺序推进：

1. 定义 `v0` schema 与工具协议
2. 打通会话与 AST 主链路
3. 补齐预览
4. 补齐修订与确认
5. 补齐导出
6. 增加回归样例与稳定性检查

详细执行计划见 `BUILD_PLAN_V0.md`。

## 10. 结论

从技术角度看，Spec Design MCP 适合采用“先收敛再扩展”的方式推进。若以 `TSD_V0.md` 为基线，本项目适合用 vibe coding 方式完成首版可用成品；若继续按原始大而全范围推进，则更适合做原型而非稳定产品。
