# Spec Design MCP PRD

## 1. 产品概述

Spec Design MCP 是一个面向外部 AI Agent / AI 工作流的 MCP 服务，用于把多模态设计需求转化为结构化 UI 设计产物。它不是人类优先的拖拽设计器，而是一个能够被 Agent 调用的“设计规划与交付能力层”。

产品完成后，调用方应能够通过一次或多次 MCP 工具调用，完成以下流程：

1. 提交文本、图片、参考网址等设计输入。
2. 让系统主动追问并澄清缺失需求。
3. 生成基于 DOM 结构的设计 AST。
4. 将 AST 渲染为可视化预览，供用户评审设计是否合适。
5. 通过自然语言问答持续修订设计。
6. 确认版本并导出 `artifact-manifest.json`、HTML、CSS、`annotation-manifest.json`、`data-ai-*` 工程标注和 `binding.schema.json`。

## 2. 背景与问题

现有 AI 编码流程普遍存在三个问题：

1. 上游设计输入非结构化。Agent 收到的是自然语言、截图、灵感链接和零散约束，难以稳定转成可执行前端方案。
2. 现有设计工具以人为中心。传统 UI 工具强调拖拽、画布和人工编辑，不适合在 Agent 工作流中作为标准化中间层。
3. 输出物与工程脱节。截图、视觉稿、参考站和半结构化描述都很难直接指导后续编码 Agent 完成高质量实现。

因此需要一个 MCP-first 的设计服务，专门解决“从多模态需求到工程可消费设计产物”的结构化转换问题。

## 3. 产品定位

### 3.1 一句话定位

一个通过会话驱动生成结构化 UI 设计稿，并在确认后输出 HTML、工程标注与 JSON Schema 契约的 MCP 服务。

### 3.2 长期愿景

成为 AI 原生开发流程中的设计中间层，让任意编码 Agent 在动手写前端之前，先拿到可追踪、可修订、可编译、可继续自动化消费的设计契约。

### 3.3 当前版本定位

先完成单页设计任务的闭环能力，优先服务 Landing Page、Dashboard、Form Page 三类页面。

## 4. 目标用户

### 4.1 核心用户

AI Agent / AI 工作流

典型角色：

- IDE 内编码 Agent
- 自动化产品生成工作流
- 多 Agent 编排系统
- 用于生成前端交付物的内部工具链

核心诉求：

- 可编程调用
- 结构化输出
- 可追踪状态
- 可持续修订
- 可继续驱动编码

### 4.2 次级用户

产品经理、设计师、前端工程师

他们不是一线调用者，但会消费该系统生成的设计产物，或通过上游工作流间接影响输入和确认。

## 5. 目标与非目标

### 5.1 产品目标

1. 支持文本、图片、参考网址等多模态输入，并统一抽象为设计意图。
2. 通过会话式追问补齐缺失信息，避免直接猜测。
3. 生成基于 DOM 结构的设计 AST，而不是图像化设计稿。
4. 在设计确认前提供可视化预览，帮助用户判断设计是否符合预期。
5. 支持通过自然语言持续修订设计版本。
6. 输出 `artifact-manifest.json`、HTML、CSS、`annotation-manifest.json`、`data-ai-*` 工程标注与 JSON Schema。
7. 让输出结果能直接作为后续编码 Agent 的输入契约。

### 5.2 非目标

1. 不做拖拽式编辑器。
2. 不做自由画布工具。
3. 不允许通过可视化预览层直接修改结构或样式。
4. 不在 MVP 阶段提供复杂交互动效编辑器。
5. 不在 MVP 阶段处理多页面站点编排。
6. 不在 MVP 阶段直接输出完整 React/Vue 应用代码。

## 6. 产品原则

### 6.1 可视化预览是必须的

虽然本产品是面向 AI Agent 的 MCP 服务，但在需求确认阶段，仍然必须向用户提供可视化预览能力。否则用户无法判断当前设计是否符合预期，也无法给出有效修订意见。

### 6.2 可视化预览不等于可视化编辑

预览层的职责是展示当前设计版本，而不是提供拖拽、框选、画布编辑等交互。所有修改仍然必须通过自然语言问答完成。

### 6.3 Schema 属于确认后交付

`binding.schema.json` 和工程标注属于设计确认后的开发契约，用于指导下游编码 Agent。它们不是设计确认阶段的主要评审载体。

## 7. 核心使用场景

### 场景 A：AI Agent 接收产品需求后生成首版设计

输入：

- 产品文案
- 品牌约束
- 参考网站
- 风格图片

流程：

1. Agent 创建设计会话。
2. Agent 追加文本、图片、URL 输入。
3. MCP 服务判断信息是否足够，不足则返回追问。
4. Agent 补齐回答后触发生成。
5. 服务返回 AST、结构摘要和可视化预览。

结果：

- Agent 拿到可继续修改的结构化设计版本。
- 用户可以直接查看预览并判断方向是否正确。

### 场景 B：AI Agent 根据反馈持续修订

输入：

- “首屏更简洁”
- “加入客户案例模块”
- “按钮更偏企业级风格”

流程：

1. Agent 调用修订工具。
2. 服务定位受影响节点并修改 AST。
3. 返回新版本、节点级变更信息、diff 摘要和更新后的可视化预览。

结果：

- Agent 可以在不进入画布编辑的前提下完成设计迭代。
- 用户可以通过连续预览比较设计是否更接近期望。

### 场景 C：为后续编码 Agent 输出工程契约

输入：

- 已确认设计版本

流程：

1. Agent 确认版本。
2. Agent 触发导出。
3. 服务返回 `artifact-manifest.json`、HTML、CSS、`annotation-manifest.json` 和 `binding.schema.json`。

结果：

- 下游编码 Agent 可以按照标注和 schema 继续实现页面。

### 场景 D：用户在确认前评审设计是否合适

输入：

- 当前设计版本

流程：

1. Agent 请求系统渲染当前设计版本。
2. 系统返回可视化预览、截图或预览地址。
3. 用户查看预览后，用自然语言表达修改意见。
4. Agent 将意见转为修订指令继续迭代。

结果：

- 用户在不使用拖拽编辑器的情况下，仍然可以完成视觉评审和方向确认。

## 8. 功能需求

### FR-001 会话创建

系统必须支持创建独立设计会话，并返回可追踪的 `sessionId`。

### FR-002 多模态输入接入

系统必须支持文本、图片、参考网址三类输入，并能持续向同一会话追加。

### FR-003 输入追踪

系统必须记录输入来源、解析结果和使用痕迹，保证参考信息可追溯。

### FR-004 结构化意图建模

系统必须将输入统一转为 `IntentModel`、`ReferenceModel[]`、`ConstraintModel`。

### FR-005 需求澄清

当信息不足或存在冲突时，系统必须返回明确追问，而不是直接生成设计。

### FR-006 设计生成

系统必须生成基于 DOM 的 `DesignDOMAST`，作为唯一设计真源。

### FR-007 设计预览

系统必须能够将当前 `DesignDOMAST` 渲染为可视化预览，用于确认前评审。

预览输出至少应包含以下一种或多种形式：

- `preview.html`
- 预览地址
- 页面截图
- 区块级视觉摘要

### FR-008 设计修订

系统必须支持自然语言驱动的 AST 修订，并生成新版本。

### FR-009 版本比较

系统必须返回修订摘要与节点级变更信息，方便外部 Agent 理解变更。

### FR-010 预览更新

每次生成或修订后，系统必须同步更新当前设计版本对应的可视化预览。

### FR-011 设计确认

系统必须允许外部 Agent 确认某一设计版本，并冻结为导出基线。

### FR-012 工程交付

系统必须导出：

- `artifact-manifest.json`
- HTML
- CSS
- `data-ai-*` 标注
- `binding.schema.json`
- `annotation-manifest.json`

### FR-013 Schema 校验

所有中间模型和导出产物都必须经过 schema 校验。

## 9. 非功能需求

### NFR-001 结构化优先

所有工具返回值必须以结构化 JSON 为主，自由文本只可作为辅助说明。

### NFR-002 一致性

预览、修订、确认、导出必须基于同一份 AST。

### NFR-003 可恢复

当模型输出非法、编译失败或多模态解析失败时，系统必须返回可操作错误，而不是静默失败。

### NFR-004 可观测

系统必须可记录会话状态、追问次数、版本数、导出成功率和 schema 校验失败原因。

### NFR-005 可扩展

架构必须支持后续接入更多页面类型、更多导出目标和更复杂的设计节点。

## 10. 关键产物定义

### 10.1 设计中间产物

- `IntentModel`
- `ReferenceModel[]`
- `ConstraintModel`
- `DesignDOMAST`

### 10.2 确认前评审产物

- `preview.html`
- `preview.png` 或其他截图产物
- `section-summary.json`
- `revision-diff.json`
- `node-diffs.json`
- `preview-metadata.json`

### 10.3 导出产物

- `compiled.html`
- `compiled.css`
- `data-ai-*` 标注
- `binding.schema.json`
- `annotation-manifest.json`
- `artifact-manifest.json`

### 10.4 下游价值

`binding.schema.json`、`annotation-manifest.json` 和 `data-ai-*` 标注共同构成后续编码 Agent 的关键开发契约。它们共同定义页面中哪些内容需要数据填充、字段类型是什么、哪些节点可能抽象成组件，以及节点的工程语义。

## 11. MVP 范围

### 11.1 MVP 必做

1. 提供 MCP Server 能力。
2. 支持文本、图片、网址三类多模态输入。
3. 支持需求澄清与追问。
4. 生成 DOM AST 初稿。
5. 生成可视化预览供确认前评审。
6. 支持问答式修订。
7. 支持确认与导出。
8. 输出 `artifact-manifest.json`、HTML、CSS、`annotation-manifest.json`、`data-ai-*` 标注和 `binding.schema.json`。

## 12. 后续路线图

### 阶段 1：MVP 闭环

目标：

- 打通从输入到导出的完整链路

交付：

- MCP 工具集
- 会话状态机
- 多模态分析
- AST 生成、预览与修订
- `artifact-manifest.json`、HTML、CSS、`annotation-manifest.json`、`data-ai-*` 标注与 `binding.schema.json` 导出

### 阶段 2：能力增强

目标：

- 提升页面复杂度、结构理解和导出质量

交付：

- 更丰富页面类型
- 更精细组件语义
- 更强版本 diff
- 更完整响应式策略

### 阶段 3：工程深度集成

目标：

- 让该服务更深地进入研发流水线

交付：

- React/Vue 映射策略
- 组件库对齐
- 数据接口契约生成增强
- 与代码生成 Agent 的联动协议

### 阶段 4：人机协同增强

目标：

- 在不改变 MCP-first 定位的前提下，增加人工审核能力

交付：

- Web 控制台
- 设计版本浏览
- 审核与批准流

## 13. 成功指标

### 13.1 产品指标

- 会话成功生成率
- 预览生成成功率
- 会话确认率
- 导出成功率
- 单会话平均追问次数
- 修订后继续被采用的版本比例

### 13.2 工程指标

- schema 校验通过率
- HTML 编译成功率
- 预览渲染成功率
- 标注完整率
- `binding.schema.json` 可被下游 Agent 成功消费的比例

## 14. 验收标准

满足以下条件即可视为 MVP 达标：

1. 外部 Agent 可通过 MCP 创建会话并追加多模态输入。
2. 服务能在信息不足时返回结构化追问。
3. 服务能生成合法的 `DesignDOMAST`。
4. 服务能生成当前设计版本对应的可视化预览。
5. 服务能依据自然语言修订生成新版本，并同步更新预览。
6. 服务能确认指定版本并完成导出。
7. 导出包包含 `artifact-manifest.json`、HTML、CSS、`annotation-manifest.json`、`data-ai-*` 标注和 `binding.schema.json`。
8. 所有中间模型与最终产物可通过 schema 校验。

## 15. 依赖与风险

### 15.1 核心依赖

- 多模态模型能力
- URL 内容抓取与解析能力
- AST 设计与编译器实现质量
- Schema 设计完整性

### 15.2 主要风险

1. 多模态输入质量不稳定，导致意图提取误差。
2. 自然语言修订指令可能存在歧义，造成 AST 修改范围不准。
3. 预览渲染若与最终导出不一致，会削弱确认环节的可靠性。
4. HTML 与 Schema 输出若不稳定，下游 Agent 将难以消费。

### 15.3 风险应对

1. 通过追问机制降低信息不完整风险。
2. 通过版本 diff 和节点级定位降低误改风险。
3. 通过“同一 AST 驱动预览与导出”降低预览偏差。
4. 通过严格 schema 校验提升产物稳定性。

## 16. 产品结论

Spec Design MCP 的核心价值不是替代传统设计工具，而是为 AI 原生研发流程补上一层结构化设计能力。MVP 的关键不是功能堆叠，而是打通一条稳定链路：多模态输入 -> 澄清 -> AST -> 可视化预览 -> 修订 -> 确认 -> 开发交付导出。只要这条链路稳定，下游编码 Agent 的质量与可控性就会显著提升。
