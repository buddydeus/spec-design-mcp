# Spec Design MCP PRD Overview

## 1. 文档目的

本文件不再承担“直接开发基线”的职责，而是作为 Spec Design MCP 的产品总览与版本分层入口。

本仓库的产品文档拆分为两层：

- 战略层：说明产品定位、长期方向、版本边界
- 执行层：定义当前可直接进入开发的范围、约束与验收

当前默认开发目标为 `v0`，以“适合 vibe coding、可较快产出可用成品”为第一优先级。

## 2. 产品定位

Spec Design MCP 是一个面向外部 AI Agent / AI 工作流的 MCP 服务，用于把非结构化设计输入转化为结构化页面设计产物，并在确认后输出后续开发可消费的交付包。

它的核心价值不是替代设计工具，而是为 AI 原生研发流程提供一个稳定的设计中间层。

## 4. 版本分层

### 4.1 V0：可用成品优先

`v0` 目标是做出一个真实可用、可演示、可继续自动化消费的最小闭环。

范围收敛为：

- 单页 `Landing Page`
- 输入以 `text` 为主，`url` 为增强能力
- 生成严格受限的 `DesignDOMAST`
- 输出可查看预览
- 支持自然语言修订
- 支持确认版本
- 支持导出 `artifact-manifest.json`、HTML、CSS、`annotation-manifest.json`、基础版 `binding.schema.json`

### 4.2 V1：能力扩展

`v1` 再逐步扩展到以下方向：

- `Dashboard`
- `Form Page`
- 图片输入作为主链路能力
- 更完整的节点语义与 diff
- 更强的数据绑定模型
- 更严格的资源治理与临时预览生命周期

### 4.3 V2：平台化能力

在产品验证通过后，再考虑：

- S3/对象存储
- 临时资源回收 worker
- 短时签名 URL
- 多 Agent 并发治理
- 审批流
- Web 控制台

## 5. 核心问题与产品决策

### 5.1 已识别的问题

第一轮评审确认了以下关键问题：

- 原始 MVP 范围过宽，不适合直接用 vibe coding 稳定落地
- 多模态、预览治理、导出契约、并发版本控制混在一版内，复杂度过高
- “愿景文档”和“开发基线文档”未分离，导致研发目标不清晰

### 5.2 已确认的产品决策

基于评审，当前产品决策如下：

1. 先做 `v0`，只追求单页闭环可用。
2. `v0` 不把图片理解、签名 URL、清理队列、平台级治理作为必须项。
3. `v0` 输出必须先保证“稳定、可读、可被下游 Agent 消费”，再追求复杂能力。
4. 所有后续增强能力都必须在 `v0` 稳定后再进入排期。

## 6. 产品目标

### 6.1 总目标

让一个外部 Agent 能够通过 MCP 完成以下链路：

1. 创建设计会话
2. 追加需求输入
3. 获取澄清问题
4. 生成首版结构化设计
5. 查看预览并给出反馈
6. 持续修订
7. 确认版本
8. 导出开发交付物

### 6.2 当前衡量标准

对于当前版本，成功不再等同于“功能很多”，而是满足以下判断：

- 有稳定的主链路
- 产物结构清晰且可验证
- 下游 Agent 无需猜测主要文件含义
- 人类可以通过预览判断设计是否可接受

## 7. 非目标

当前不追求以下目标：

- 替代传统设计器
- 覆盖所有页面类型
- 一次性解决完整前端工程生成
- 首版就实现平台级资源治理
- 首版就支持多人协同审核流

## 8. 总路线图

### 阶段 A：V0 可用闭环

- 单页 Landing Page
- 文本优先输入
- URL 增强解析
- AST -> 预览 -> 修订 -> 确认 -> 导出

### 阶段 B：V1 范围扩展

- Dashboard / Form Page
- 图片输入
- 更细粒度 diff
- 更强绑定 schema

### 阶段 C：V2 平台化

- 对象存储
- 签名 URL
- Cleanup Worker
- 外部控制台
- 审批与多角色协同

## 9. 结论

Spec Design MCP 的方向成立，但它必须先从一个收敛、明确、可被 vibe coding 可靠推进的 `v0` 开始。后续版本应建立在 `v0` 的真实使用反馈之上，而不是在首版中同时引入过多系统复杂度。
